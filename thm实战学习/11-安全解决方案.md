# SIEM
安全信息和事件管理（Security Information and Event Management system）系统是安全运营中心（SOC）分析师使用的核心安全解决方案。在本课程中，我们将学习网络中的不同设备如何生成日志，以及为什么拥有一个集中式解决方案来收集、规范化和关联这些日志至关重要。
## 日志源以及分析困难
网络中的设备会持续生成记录其内部活动的日志。我们也可以将这些设备称为**日志源**。它们生成的日志记录了所有活动的轨迹，对于识别恶意活动或进行常规故障排除极其有用。这些日志源主要分为以下两大类：

### 1. 以主机为中心的日志源 (Host-Centric Log Sources)
这类日志源记录发生在主机内部或与主机相关的事件。生成此类日志的设备包括 Windows、Linux 系统及服务器等。常见示例包括：
* 用户访问文件
* 用户尝试身份验证（登录）
* 进程执行活动
* 进程添加/编辑/删除注册表键值
* PowerShell 脚本执行

### 2. 以网络为中心的日志源 (Network-Centric Log Sources)
当主机之间相互通信或访问互联网网站时，会产生网络相关日志。生成此类日志的设备包括防火墙、IDS/IPS（入侵检测/防御系统）、路由器等。常见示例包括：
* SSH 连接
* 通过 FTP 协议访问文件
* Web 流量
* 用户通过 VPN 访问公司资源
* 网络文件共享活动

这些主机和网络日志源共同在网络中不断产生大量的日志记录。

---

### 面临的挑战
到目前为止，逻辑似乎很简单：日志源生成日志，我们进行分析并识别恶意活动。然而，实际操作并没那么简单，它面临着以下挑战：

* **日志源数量庞大：** 一个网络拥有众多日志源，每秒会产生数百个事件。这些日志散落在不同的设备上，在发生安全事件时逐一检查每台设备会非常乏味且低效。
* **缺乏集中管理：** 由于日志保存在生成它们的机器上，你可能需要通过 SSH、RDP 等方式连接到每个日志源来分析数据。这在调查过程中非常低效，会浪费大量宝贵时间。
* **上下文关联有限：** 单条日志无法提供活动的完整全貌。在安全事件中，不同日志源上的单个活动看似无害，但如果将这些日志关联起来，可能会揭示完全不同的真相。
* **分析能力受限：** 日志源每秒产生海量日志，人工手动分析所有设备的日志以识别异常几乎是不可能的。现实中，分析师会因为数据量过于巨大而漏掉许多重要的关键日志。
* **格式不统一：** 不同的日志源生成的日志格式各异。分析师需要熟悉所有这些格式才能进行分析，当面对网络中众多的日志源时，这变得极其困难。

在前一任务中，我们了解了不同日志源如何生成海量且多样的日志，以及分析这些日志所面临的挑战。那么，我们该如何更高效地管理这些数据洪流并提取有价值的信息呢？

这正是 **SIEM** 发挥作用的地方。**安全信息和事件管理（SIEM）** 是一种安全解决方案，它从各种类型的日志源收集日志，将其格式标准化，进行关联分析，并利用检测规则发现恶意活动。

---

## SIEM 的核心功能
### 1. 集中式日志收集 (Centralized Log Collection)
SIEM 从所有来源（终端、服务器、防火墙等）收集日志并汇总到一处。这些日志通过轻量级代理（Agents）或 **API** 拉取并填充到 SIEM 方案中。这解决了必须逐一登录每台机器分析日志的难题。

### 2. 日志规范化 (Normalization)
原始日志的格式和大小各不相同（例如 Windows 日志与 Linux 日志截然不同）。
* **解析 (Parsing)：** 将一条日志拆分成多个易于理解的字段（如时间、用户、IP）。
* **规范化 (Normalization)：** 将来自不同源的日志转换为统一的一致格式，确保在同一平台上可以进行对比和检索。

### 3. 日志关联 (Correlation)
单条日志的价值有限。SIEM 会关联不同源的日志并寻找它们之间的联系，通过分析模式来识别恶意活动。
> **案例分析（5分钟内发生的活动）：**
> 1. 用户 Haris 从一个从未用过的 IP 通过 VPN 登录。
> 2. Haris 访问了共享驱动器上的某些文档。
> 3. Haris 执行了一个 PowerShell 脚本。
> 4. 系统发起了一个出站网络连接。
>
> 单独看这些行为可能没问题，但 SIEM 关联后会发现：这极有可能是 Haris 的 VPN 凭据泄露后导致的数据外泄行为。

### 4. 实时告警 (Real-time Alerting)
SIEM 根据内置规则检测恶意活动。
* **默认规则：** 随系统附带。
* **自定义规则：** 分析师根据需求创建，以完善检测能力。
当触发规则条件时，系统会产生**告警（Alerts）**并通知分析师，分析师随后在 SIEM 平台内进行调查。

### 5. 仪表盘与报告 (Dashboards and Reporting)
仪表盘是 SIEM 最直观的组件，将规范化后的数据转化为**可视化洞察**。常见展示内容包括：
* 告警摘要与系统通知
* 健康状态警报
* 登录失败列表
* 日志摄入总量及触发规则统计
* 访问量最高的域名

---

## 其他高级功能
除了上述核心功能，SIEM 通常还具备：
* **威胁情报集成：** 结合外部已知的恶意 IP 或文件黑名单。
* **海量数据保留：** 用于满足合规性或长期追溯。
* **强大的搜索能力：** 支持从数以亿计的记录中快速定位信息。

## 日志源与接入 (Log Sources and Ingestion)

### 1. 日志源 (Log Sources)
网络中的每台设备在执行活动（如访问网站、SSH 连接、登录工作站）时都会生成日志。以下是网络环境中常见设备的日志形式：

#### **Windows 机器**
Windows 系统记录的所有事件都可以通过**事件查看器 (Event Viewer)** 查看。
* **特点：** Windows 为每种类型的日志活动分配唯一的 **ID**（即 Event ID），方便分析师追踪。
* **查看方式：** 在搜索栏输入 "Event Viewer" 即可打开该工具。在企业环境中，这些日志通常会从终端转发到 SIEM 以实现集中监控。

#### **Linux 机器**
Linux 操作系统存储了包括事件、错误、警告在内的所有相关日志。常见的日志存储路径包括：
* `/var/log/httpd`: 包含 HTTP 请求/响应和错误日志。
* `/var/log/cron`: 存储与定时任务 (Cron Jobs) 相关的事件。
* `/var/log/auth.log` 或 `/var/log/secure`: 存储身份验证相关的日志（如登录尝试）。
* `/var/log/kern`: 存储内核相关的事件。
> **Cron 日志示例：** 记录了守护进程的启动以及各项定时任务的状态。

#### **Web 服务器**
监控 Web 服务器的所有进出请求对于识别 Web 攻击至关重要。
* **存储位置：** 在 Linux 中，Apache 相关的日志通常位于 `/var/log/apache` 或 `/var/log/httpd`。
* **日志内容：** 包含访问者 IP、访问时间、请求方法（GET/POST）、请求路径、响应状态码（如 200）以及浏览器标识（User-Agent）。

---

### 2. 日志接入 (Log Ingestion)
虽然日志包含丰富的信息，但必须先进入 SIEM 系统才能发挥作用。不同的 SIEM 方案有不同的接入方法，常见方式如下：

| 接入方式 | 说明 |
| :--- | :--- |
| **代理/转发器 (Agent / Forwarder)** | 在终端安装轻量级工具（如 Splunk 的 Forwarder）。它负责抓取重要日志并将其主动发送到 SIEM 服务器。 |
| **系统日志协议 (Syslog)** | 一种广泛使用的标准化协议，用于从 Web 服务器、数据库等系统实时向集中目的地发送数据。 |
| **手动上传 (Manual Upload)** | 许多 SIEM（如 Splunk, ELK）支持手动上传脱机数据。数据上传后会被立即规范化并用于快速分析。 |
| **端口转发 (Port-Forwarding)** | SIEM 监听特定端口，终端设备则配置为直接将数据流推送到 SIEM 实例的该监听端口。 |

## 告警触发的幕后原理
### 1. 什么是检测规则？
SIEM 依靠**检测规则 (Detection Rules)** 来捕捉威胁。这些规则本质上是预设的**逻辑表达式**，旨在确保分析师能及时采取行动。

**常见的规则示例：**
* **暴力破解检测：** 如果 10 秒内出现 5 次登录失败，触发“多次登录失败”告警。
* **暴破成功检测：** 如果在多次登录失败后出现一次登录成功，触发“多次失败后登录成功”告警。
* **合规性监控：** 每当用户插入 USB 设备时发出告警（适用于公司禁止使用私有存储介质的政策）。
* **数据外泄检测：** 如果出站流量超过 25 MB，触发“潜在数据外泄”告警（具体阈值取决于公司政策）。

### 2. 检测规则是如何创建的？
为了理解规则的工作方式，我们来看两个基于 Windows 事件日志的案例：

#### **案例 1：日志清除行为**
攻击者在利用漏洞（Post-Exploitation）后，通常会删除日志以抹除踪迹。Windows 中，每当用户尝试清除事件日志时，都会记录一个唯一的 **Event ID 104**。
* **规则逻辑：** 如果 `日志源 == WinEventLog` 且 `EventID == 104` $\rightarrow$ 触发告警：**“事件日志已清除”**。

#### **案例 2：可疑命令执行**
攻击者在提权后常会使用 `whoami` 命令来确认当前权限。
* **关键字段：**
    * **日志源：** WinEventLog
    * **事件 ID：** 4688（表示进程创建活动）
    * **进程名称：** whoami
* **规则逻辑：** 如果 `日志源 == WinEventLog` 且 `EventID == 4688` 且 `NewProcessName 包含 whoami` $\rightarrow$ 触发告警：**“检测到 WHOAMI 命令执行”**。

> **注意：** 这就是为什么**日志规范化**如此重要——规则需要监控特定字段（如 EventID 或 ProcessName）的准确取值才能生效。

### 3. 告警调查 (Alert Investigation)
在日常工作中，SOC 分析师大部分时间都在查看仪表盘。一旦告警触发，分析师会检查相关的事件流和触发条件，判断其是**误报**还是**真报**。

**分析后的后续动作通常包括：**
* **误报 (False Positive)：** 该告警是误报。可能需要微调（Tuning）规则逻辑，以防止类似误报再次发生。
* **真报 (True Positive)：** 该告警是真实威胁。需执行进一步调查：
    * 联系资产负责人核实活动是否为授权操作。
    * 如果确认是恶意活动：**隔离受感染主机**。
    * 在防火墙上**阻断可疑 IP**。

# 防火墙基础
**防火墙**是位于内部网络与外部互联网之间的数字“保安”，它通过预设的安全规则对进出流量进行实时检查，充当拦截非法访问、放行授权通信的核心防御屏障，确保未经授权的威胁无法通过网络边界潜入系统。
随着组织意识到防火墙过滤有害流量的能力，各种针对不同场景的防火墙应运而生。需要注意的是，不同类型的防火墙工作在 **OSI 模型**的不同层级。

以下是四种最常见的防火墙类型及其在 OSI 模型中的角色：

---

### 1. 无状态防火墙 (Stateless Firewall)
* **工作层级：** 第 3 层（网络层）和 第 4 层（传输层）。
* **工作原理：** 仅根据预设规则（如 IP、端口）过滤数据，**不记录**之前连接的状态。
* **特点：** 每个数据包都被视为独立的，即使它是合法连接的一部分，也要重新匹配规则。
* **优缺点：** 处理速度极快，适用于高速网络；但无法基于历史连接应用复杂的安全策略（例如：它会“忘记”刚才被拦截的恶意源，对该源后续发来的包仍需重新检查）。

### 2. 有状态防火墙 (Stateful Firewall)
* **工作层级：** 第 3 层和 第 4 层。
* **工作原理：** 除了预设规则外，还会通过**状态表（State Table）**跟踪并记录之前的连接。
* **特点：** 如果一个连接的首个数据包被允许，后续包将自动放行而无需再次检查。反之，如果之前的包被拒绝，后续包也会被直接拦截。
* **优势：** 相比无状态防火墙，增加了基于“连接历史”的安全维度。

### 3. 代理防火墙 (Proxy Firewall)
* **工作层级：** 第 7 层（应用层），也称为应用层网关。
* **工作原理：** 充当私有网络与互联网之间的**中间人**。它会拆解并检查数据包的实际**内容**。
* **特点：** 它会用自己的 IP 掩盖内部真实 IP（提供匿名性），并能基于内容过滤策略（如拦截包含特定关键字的网页）来控制流量。

### 4. 下一代防火墙 (NGFW)
* **工作层级：** 第 3 层到 第 7 层。
* **工作原理：** 目前最先进的防火墙，提供深度包检测（DPI）及多种增强安全功能。
* **核心功能：** * **入侵防御系统 (IPS)：** 实时阻断恶意活动。
    * **启发式分析：** 通过分析攻击模式，在威胁到达网络前识别并拦截。
    * **SSL/TLS 解密：** 能够解密加密流量进行检查，并结合**威胁情报**做出高效决策。


### 防火墙特性对比表

| 防火墙类型 | 核心特性 |
| :--- | :--- |
| **无状态防火墙** | 基础过滤；不跟踪连接；适用于高速网络环境。 |
| **有状态防火墙** | 基于模式识别流量；可应用复杂规则；监控网络连接状态。 |
| **代理防火墙** | **检查包内数据内容**；支持内容过滤与应用控制；解密并检查 SSL/TLS 数据。 |
| **下一代防火墙 (NGFW)** | 提供高级威胁保护；**内置 IPS**；基于启发式分析识别异常；解密并结合情报分析。 |

## 防火墙规则
### 1. 防火墙规则的基本组件
每一条防火墙规则都由以下几个核心要素构成：
* **源地址 (Source Address)：** 发起流量的机器 IP 地址。
* **目的地址 (Destination Address)：** 接收数据的机器 IP 地址。
* **端口 (Port)：** 流量所使用的特定端口号。
* **协议 (Protocol)：** 通信所使用的协议（如 TCP、UDP）。
* **动作 (Action)：** 当流量匹配规则时，防火墙采取的具体操作。
* **方向 (Direction)：** 定义该规则适用于“进入”还是“发出”的流量。

---

### 2. 动作类型 (Types of Actions)
当数据包匹配规则时，防火墙通常会执行以下三种主要动作之一：

#### **允许 (Allow)**
放行符合规则定义的流量。
* **示例：** 允许内网所有机器通过 80 端口访问互联网（HTTP 流量）。

| 动作 | 源地址 | 目的地址 | 协议 | 端口 | 方向 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **Allow** | 192.168.1.0/24 | Any | TCP | 80 | Outbound (出站) |

#### **拒绝 (Deny)**
阻断并禁止符合规则定义的流量。这是减少网络威胁暴露面的基本手段。
* **示例：** 禁止任何外部 IP 通过 22 端口（SSH）连接到内网的关键服务器。

| 动作 | 源地址 | 目的地址 | 协议 | 端口 | 方向 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **Deny** | Any | 192.168.1.0/24 | TCP | 22 | Inbound (入站) |

#### **转发 (Forward)**
将流量重定向到不同的网络段。这常见于具备路由功能的防火墙或网关。
* **示例：** 将所有进入 80 端口的流量转发到内网真正的 Web 服务器（192.168.1.8）。

| 动作 | 源地址 | 目的地址 | 协议 | 端口 | 方向 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **Forward** | Any | 192.168.1.8 | TCP | 80 | Inbound (入站) |

---

### 3. 规则的方向性 (Directionality)
根据流量的流向，规则通常被归纳为以下三类：

* **入站规则 (Inbound Rules)：** 仅适用于**进入**网络的流量。
* **出站规则 (Outbound Rules)：** 仅适用于从网络内部**发出**的流量。
* **转发规则 (Forward Rules)：** 用于在网络内部不同网段之间**传递**特定流量。

## Windows防火墙

### 1. 简介
**Windows Defender 防火墙**是 Windows 操作系统内置的安全组件。它提供了创建、允许或拒绝特定程序以及自定义规则的基础功能，通过限制系统的进出流量来保护安全。

---

### 2. 网络配置文件 (Network Profiles)
Windows 会根据 **网络位置感知 (NLA)** 自动判断当前所处的网络环境，并应用相应的配置：

* **专用网络 (Private networks)：** 适用于信任的环境，如家庭或办公室网络。
* **公用网络 (Guest or public networks)：** 适用于咖啡店、机场等不可信的公共场所。
    * *安全策略建议：* 在公共网络下，通常建议配置为“阻止所有入站连接”，仅允许必要的出站连接。

**基础操作：**
1.  **允许应用：** 点击“允许应用或功能通过 Windows Defender 防火墙”，可手动勾选允许在特定模式下运行的程序。
2.  **开启/关闭：** 虽可关闭，但微软强烈建议保持开启，或仅选择“阻止所有传入连接”。
3.  **恢复默认值：** 若配置错误，可随时点击“恢复默认设置”。

---

### 3. 创建自定义规则 (Custom Rules)
为了实现更精细的控制（例如禁止访问网页），我们需要通过**高级设置**来创建规则。

#### **实战案例：阻止所有出站 HTTP (80) 和 HTTPS (443) 流量**
目标：创建规则后，系统将无法通过浏览器访问任何互联网网站。

**操作步骤：**
1.  **进入界面：** 在主面板点击“高级设置”，选择左侧的“出站规则”，然后点击右侧的“新建规则”。
2.  **规则类型：** 选择“自定义 (Custom)”。
3.  **程序：** 选择“所有程序 (All programs)”。
4.  **协议与端口：**
    * 协议类型：**TCP**
    * 远程端口：选择“特定端口”，输入 **80, 443**（注意：中间用逗号分隔，不要加空格）。
5.  **作用域 (Scope)：** 保持默认（适用于所有本地和远程 IP）。
6.  **操作 (Action)：** 选择“阻止连接 (Block the connection)”。
7.  **配置文件：** 勾选所有配置文件（域、专用、公用）。
8.  **命名：** 给规则起一个名字（如 "Block_Web_Traffic"）并完成。

**测试结果：**
* **规则前：** 访问 `http://10.10.10.10/` 正常。
* **规则后：** 访问该地址会显示“无法访问此页面”，证明规则已生效。

## Linux防火墙
如果你是 Linux 用户，同样需要控制网络流量。Linux 提供了一套强大的内置防火墙功能，其核心架构和多种管理工具可以满足不同层次的需求。

---

### 1. 核心框架：Netfilter
**Netfilter** 是 Linux 内核中的一个框架，它提供了包过滤、网络地址转换（NAT）和连接跟踪等核心功能。它是 Linux 所有防火墙工具的**基石**。

以下是基于 Netfilter 开发的常见管理工具：
* **iptables：** 最经典、使用最广泛的工具。虽然功能强大，但规则语法相对复杂。
* **nftables：** iptables 的继任者，优化了性能并增强了数据包过滤能力。
* **firewalld：** 常见于 CentOS/RHEL 体系。它使用“区域（Zones）”的概念来管理规则，更加动态化。

---

### 2. 简易工具：UFW (Uncomplicated Firewall)
正如其名，**UFW** 旨在消除 iptables 复杂的语法，为用户提供一个极其简单的命令行界面。它非常适合初学者，你只需输入简单的指令，UFW 就会在后台自动将其转换为复杂的 iptables 规则。

#### **常用 UFW 命令：**

* **查看状态：**
    `sudo ufw status`
    *(如果显示 `inactive`，说明防火墙尚未开启)*

* **启用/禁用防火墙：**
    启用：`sudo ufw enable`
    禁用：`sudo ufw disable`

* **设置默认策略：**
    你可以设置一个全局的默认行为。例如，允许所有出站流量：
    `sudo ufw default allow outgoing`

* **拒绝特定服务（以 SSH 为例）：**
    如果你想阻止 22 端口（SSH）的入站连接，可以使用：
    `sudo ufw deny 22/tcp`
    *(此处指定了操作：`deny`，以及协议/端口：`22/tcp`)*

* **按编号列出规则：**
    为了方便管理，可以查看带编号的规则列表：
    `sudo ufw status numbered`

* **删除规则：**
    根据上面查到的编号（例如编号 2）来删除规则：
    `sudo ufw delete 2`

# 入侵检测系统（IDS）
如果说防火墙扮演的是门卫的角色，那么 IDS 扮演的就是监控摄像头的角色。它静静地待在角落，基于特征码或异常检测技术来监控网络流量，识别进出网络的所有异常行为。

每当检测到威胁时，IDS 都会为安全管理员生成一条告警。需要注意的是，IDS 本身不会对这些检测采取行动（如阻断），它仅仅是通知管理员发生了恶意活动。

## IDS类型
IDS 可以根据不同的因素进行分类，主要分类方式取决于其**部署模式**和**检测模式**。

---

### 1. 部署模式

* **主机型入侵检测系统 (HIDS)：**
    HIDS 安装在单个主机上，仅负责检测与该特定主机相关的潜在安全威胁。它能提供对主机活动的详细可见性。然而，在大规模网络中管理 HIDS 非常具有挑战性，因为它们资源消耗大，且需要对每台主机进行单独管理。
* **网络型入侵检测系统 (NIDS)：**
    NIDS 负责检测整个网络中的恶意活动，而不针对特定主机。它通过监控所有相关主机的网络流量来发现可疑行为。它能为整个网络内部的检测提供一个集中化的视图。

---

### 2. 检测模式

* **基于特征的 IDS (Signature-Based IDS)：**
    每天都会发生大量攻击，每种攻击都有其独特的模式，称为“特征（Signature）”。IDS 会在数据库中保存这些特征。如果未来发生同样的攻击，系统就能通过匹配特征进行检测并上报。
    * **优点：** 只要数据库强大，检测已知威胁非常高效。
    * **缺点：** **无法检测零日漏洞（Zero-day）攻击**，因为零日攻击没有先前的模式保存。它只能识别数据库中已有的“陈旧”威胁。我们将在接下来的任务中学习一个名为 **Snort** 的特征检测工具。
* **基于异常的 IDS (Anomaly-Based IDS)：**
    此类 IDS 首先学习网络或系统的“正常行为（基准线）”。一旦当前状态偏离了正常范围，就会触发检测。
    * **优点：** **可以检测零日攻击**，因为它不依赖已知特征，而是通过对比基准线来发现异常。
    * **缺点：** 容易产生大量**误报（False Positives）**。由于许多合法程序的行为模式可能与恶意软件相似，系统可能会将所有不寻常的行为都视为恶意。通过手动定义正常行为（精细化调整/Fine-tuning）可以减少误报。
* **混合型 IDS (Hybrid IDS)：**
    混合型 IDS 结合了上述两种方法的优势。
    * **逻辑：** 如果是已知威胁，使用特征检测技术快速识别；如果是新威胁，则利用异常检测技术。这种方式结合了速度与广度。

## IDS 实例：Snort

**Snort** 诞生于 1998 年，是全球应用最广泛的开源 IDS（入侵检测系统） 解决方案之一。它结合了基于特征和基于异常的检测技术，通过内置的规则文件来识别已知威胁。

### 1. 核心规则机制

  * **内置规则：** Snort 软件包预装了大量的规则文件，涵盖了多种已知的攻击模式，能够自动检测多数恶意流量。
  * **自定义能力：** 用户可以根据需求编写自定义规则，以检测默认规则未涵盖的特定流量。
  * **灵活性：** 如果某些内置规则对你的系统产生了误报，你可以随时禁用它们，并定义更符合现状的规则。

-----

### 2. Snort 的三种工作模式

根据不同的使用场景，Snort 可以切换以下三种模式：

| 模式 | 描述 | 典型用例 |
| :--- | :--- | :--- |
| **数据包嗅探模式**<br>(Packet Sniffer) | 仅读取并实时显示网络数据包，**不进行分析**。这与 IDS 功能无直接关系，主要用于流量监控。 | **网络排障：** 网络团队观察到性能问题，需要查看详细的流量信息来诊断故障，此时可使用嗅探模式。 |
| **数据包记录模式**<br>(Packet Logging) | 将网络流量实时记录并保存为 **PCAP**（标准捕获格式）文件。这包括了所有流量及其检测到的告警信息。 | **数字取证：** 安全团队需要对过去的攻击进行根因分析（Root Cause Analysis），通过记录模式保存的日志文件是至关重要的证据。 |
| **网络入侵检测模式**<br>(NIDS) | **核心模式**。实时监控流量，并将其与规则库中的攻击特征进行匹配。一旦匹配成功，立即触发告警。 | **主动监控：** 安全团队需要实时防御潜在威胁，通过 NIDS 模式实现自动化的安全预警。 |

## Snort使用
Snort 的核心文件都存储在 `/etc/snort` 目录中。其中最重要的文件是 **`snort.conf`**（配置文件），你可以在此指定启用哪些规则文件、监控哪个网段以及进行其他全局设置。所有的规则文件则存放在 `rules` 文件夹下。

---

### 1. Snort 规则格式

Snort 规则有一套严谨的编写逻辑。以下是一个用于检测任何 IP 访问本地回环地址（Loopback）产生的 ICMP（Ping）包的规则示例：

`alert icmp any any -> 127.0.0.1 any (msg:"Loopback Ping Detected"; sid:10003; rev:1;)`

#### **组件详解：**
* **Action（动作）：** 触发规则后采取的操作。例如 `alert` 表示生成告警。
* **Protocol（协议）：** 匹配的协议。Ping 命令使用的是 `icmp`。
* **Source IP/Port（源 IP 与端口）：** 流量发起方的地址。`any any` 表示匹配任何 IP 和任何端口。
* **Direction（方向）：** `->` 表示流量的方向。
* **Destination IP/Port（目的 IP 与端口）：** 流量接收方。例如 `127.0.0.1` 或变量 `$HOME_NET`（在配置文件中定义的本地网段）。
* **Rule Metadata（规则元数据）：** 括号内的附加信息：
    * **msg：** 告警时显示的提示文字。
    * **sid：** 规则的唯一 ID。
    * **rev：** 规则的版本号，每次修改规则时应递增。

---

### 2. 规则创建与测试

#### **第一步：编辑自定义规则文件**
使用文本编辑器打开本地规则文件：
`sudo nano /etc/snort/rules/local.rules`
在文件末尾添加上述规则并保存。

#### **第二步：启动 Snort 进行检测**
执行以下命令让 Snort 开始监控流量：
`sudo snort -q -l /var/log/snort -i lo -A console -c /etc/snort/snort.conf`
* `-q`：静默模式，不显示标志文字。
* `-i lo`：指定监控的网卡接口（此处为回环接口）。
* `-A console`：将告警直接输出到控制台。

#### **第三步：触发告警**
在另一个终端执行 `ping 127.0.0.1`。此时，Snort 终端窗口会实时弹回如下信息：
`[**] [1:10003:1] Loopback Ping Detected [**] {ICMP} 127.0.0.1 -> 127.0.0.1`

---

### 3. 在 PCAP 文件上运行 Snort 
除了实时监控，Snort 还可以对历史流量数据（PCAP 格式）进行**取证分析**。

如果你有一个名为 `Task.pcap` 的文件，可以使用以下命令进行离线检测：
`sudo snort -q -l /var/log/snort -r Task.pcap -A console -c /etc/snort/snort.conf`
* **`-r`：** 表示读取（Read）并处理 PCAP 文件。

# 漏洞扫描器概述
## 漏洞扫描 (Vulnerability Scanning)
### 1. 自动化扫描的价值
手动寻找漏洞不仅费时费力，而且极易遗漏关键风险。
* **高效性：** 自动扫描工具极大简化了工作流，只需输入 IP 地址或网络范围，工具便会自动检测并生成易读的报告。
* **补丁 (Patches)：** 在识别出漏洞后，组织通过修改软件程序或系统来修复这些弱点，这些修复程序被称为“补丁（patches）”。

---

### 2. 扫描类型详解

#### **A. 凭据扫描 vs. 无凭据扫描 (Authenticated vs. Unauthenticated)**

| 特性 | 凭据扫描 (Authenticated) | 无凭据扫描 (Unauthenticated) |
| :--- | :--- | :--- |
| **访问要求** | 需要目标主机的登录凭据。 | 无需凭据，只需目标 IP 地址。 |
| **模拟视角** | 模拟**已进入系统**或拥有权限的攻击者。 | 模拟**外部**且没有任何权限的攻击者。 |
| **详细程度** | **更详细**。通过检查配置和安装的软件，提供系统深度的可见性。 | **较浅**。主要发现外部暴露的风险，配置更简单。 |
| **典型案例** | 提供数据库账号，扫描内网数据库的内部漏洞。 | 对外网门户网站进行黑盒扫描。 |

#### **B. 内网扫描 vs. 外网扫描 (Internal vs. External)**

| 特性 | 内网扫描 (Internal) | 外网扫描 (External) |
| :--- | :--- | :--- |
| **扫描位置** | 从网络内部发起。 | 从网络外部（互联网）发起。 |
| **侧重点** | 关注**内部扩散**风险。 | 关注**外部入侵**风险。 |
| **目标** | 识别一旦攻击者突破边界后，在内网可利用的弱点。 | 识别攻击者在进入网络前就能看到的暴露面。 |

---
## 漏洞扫描工具

### 1. Nessus
**Nessus** 最初于 1998 年作为开源项目开发，后于 2005 年由 Tenable 收购并转为专有（商业）软件。
* **地位：** 拥有极其丰富的扫描选项，被大型企业广泛使用。
* **版本：** 提供免费版和付费版。免费版功能有限；商业版则提供高级扫描、无限次扫描及专业技术支持。
* **部署：** 需要在本地环境进行部署和管理。

### 2. Qualys
**Qualys** 成立于 1999 年，是一款基于订阅制的漏洞管理解决方案。
* **功能：** 除了持续的漏洞扫描外，还提供合规性检查和资产管理，并在持续监控过程中发现漏洞时自动发出警报。
* **优势：** 它是一个**云原生平台**，这意味着用户无需在物理硬件上投入额外的成本或精力来维持系统运行和管理。

### 3. Nexpose
**Nexpose** 由 Rapid7 公司于 2005 年开发，是一款订阅制的漏洞管理方案。
* **特点：** 它能持续发现网络中的新资产并进行扫描。
* **风险评估：** 它会根据资产价值和漏洞的影响力给出“漏洞风险评分”，并针对多种标准提供合规性检查。
* **部署：** 支持本地部署及混合部署（云端 + 本地）。

### 4. OpenVAS (开放漏洞评估系统)
**OpenVAS** 是由 Greenbone Security 开发的开源漏洞评估解决方案。
* **地位：** 虽然其功能广度不如商业工具，但它提供了完整漏洞扫描器的基本体验。
* **适用场景：** 对于小型组织和个人系统非常有用。

---

### 5. 报告功能
几乎所有的漏洞扫描器都提供**报告功能**。在每次扫描后，它们会生成一份详细报告，内容包括：
* 发现的漏洞列表。
* 风险评分。
* 漏洞的详细描述。
* **高级功能：** 一些高端扫描器还提供修复建议（Remediation methods），并支持将报告导出为多种格式。

## CVE与CVSS
为了更好地理解 **CVE** 和 **CVSS**，我们可以把它们想象成 IT 服务台（Help Desk）处理投诉单的系统。每天面对成百上千的报修，你需要一种方法来标记它们，并确定处理的先后顺序。

---

### 1. CVE：漏洞的“身份证号”
**CVE** 全称是 **公共漏洞和暴露 (Common Vulnerabilities and Exposures)**。

在服务台场景中，CVE 就像是每一个投诉单的**唯一编号**。无论后续有多少次沟通或更新，只要报出这个编号，大家就知道在谈论哪一个问题。

* **定义：** 由 **MITRE** 公司开发，为每一个被发现的软件漏洞分配一个全球唯一的标识符。
* **目的：** 方便公众查阅漏洞详情，并及时采取防护措施（如打补丁）。
* **格式解析：** 例如 `CVE-2024-9374`
    * **前缀：** 始终以 `CVE` 开头。
    * **年份：** 漏洞被发现或分配编号的年份（如 2024）。
    * **序列号：** 四位或更多位数的任意数字。

---

### 2. CVSS：漏洞的“严重程度分值”
**CVSS** 全称是 **通用漏洞评分系统 (Common Vulnerability Scoring System)**。

回到服务台场景，如果你手头有 100 张投诉单，你肯定想先处理最紧急的。如果每个单子都有一个 0 到 10 的评分，分数越高代表问题越严重，你的工作就会高效得多。

* **定义：** 这是一套用于评估漏洞严重程度的标准。
* **计算方式：** 综合考虑多个因素，包括漏洞利用的难易程度、对系统的影响范围（机密性、完整性、可用性）等。
* **严重等级对照表：**

| CVSS 分值范围 | 严重等级 (Severity) |
| :--- | :--- |
| **0.0 - 3.9** | **低危 (Low)** |
| **4.0 - 6.9** | **中危 (Medium)** |
| **7.0 - 8.9** | **高危 (High)** |
| **9.0 - 10.0** | **严重 (Critical)** |

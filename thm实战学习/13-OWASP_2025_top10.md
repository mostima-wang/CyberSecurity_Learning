**OWASP Top 10 2025** 是由开放式 Web 应用程序安全项目（OWASP 基金会）发布的最新版**全球十大 Web 应用安全风险报告**。

OWASP 每几年会根据行业调查和漏洞数据更新一次这份名单。2025 版反映了现代开发环境（如云原生、AI 驱动和复杂的供应链）中安全威胁的变化。

# 2025 年十大安全风险列表

| 排名 | 风险类别 | 核心关注点 |
| :--- | :--- | :--- |
| **A01** | **失效的访问控制 (Broken Access Control)** | 依然稳居榜首。指用户越权访问了不该看的数据或功能（如普通用户查看管理员后台）。 |
| **A02** | **安全配置错误 (Security Misconfiguration)** | 排名大幅上升。涵盖了不安全的默认设置、云存储权限泄露、多余的 HTTP 标头等。 |
| **A03** | **软件供应链失败 (Software Supply Chain Failures)** | **新增/重大更新**。关注第三方库、CI/CD 流水线和包管理器的安全（如恶意组件注入）。 |
| **A04** | **加密失败 (Cryptographic Failures)** | 保护敏感数据的加密手段不足、使用过时的加密算法或密钥泄露。 |
| **A05** | **注入 (Injection)** | 排名有所下滑。包括 SQL 注入、跨站脚本 (XSS)、OS 命令注入等，主要源于对输入过滤不严。 |
| **A06** | **不安全的设计 (Insecure Design)** | 强调“安全应始于设计阶段”。如果架构本身有问题，后续写再好的代码也补不回来。 |
| **A07** | **身份验证失败 (Authentication Failures)** | 登录流程中的缺陷，例如允许弱密码、凭据填充攻击或会话管理不当。 |
| **A08** | **软件或数据完整性失败 (Integrity Failures)** | 应用程序在没有验证签名或完整性的情况下加载了不可信的代码、数据或对象。 |
| **A09** | **安全日志和记录失败 (Logging & Alerting Failures)** | 攻击发生时没有记录日志，或者报警太慢，导致黑客在系统中潜伏数月无人知晓。 |
| **A10** | **异常情况处理不当 (Mishandling of Exceptional Conditions)** | **2025版新入选**。指系统在报错时泄露敏感信息，或因未捕获异常导致拒绝服务或逻辑绕过。 |

# IAAA失败：A07、A07、A09
## IAAA
**IAAA** 是思考用户及其操作在应用程序中如何被验证的一种简单方式。其中的每一个环节都至关重要，且无法跳过任何一个层级。这意味着，如果前一个环节没有执行，你就无法进行后续的操作。这四个项分别是：

* **标识 (Identity)** —— 代表个人或服务的唯一账号（例如：用户 ID 或电子邮件）。
* **认证 (Authentication)** —— 证明该身份的真实性（例如：密码、一次性密码 OTP、通行密钥 Passkeys）。
* **授权 (Authorization)** —— 确定该身份被允许执行的操作。
* **审计/问责 (Accountability)** —— 记录并警报谁在何时、何地做了什么。

本节讨论的 **OWASP Top 10:2025** 中的三个类别，都与 **IAAA** 实施过程中的失败有关。这些环节的弱点可能具有极大的破坏性，因为它们可能允许威胁参与者访问其他用户的数据，或获得超出其应有范围的更高权限。

## A01：失效的访问控制
 失效的访问控制（Broken Access Control）发生在服务器未能针对每一个请求正确执行“谁能访问什么”的限制时。这种情况最常见的案例是 IDOR（不安全的直接对象引用）：如果你通过修改 ID（例如将 ?id=7 改为 ?id=6）就能查看或编辑他人的数据，那么访问控制就失效了。

在实际操作中，这通常表现为水平越权（相同角色，访问其他用户的数据）或垂直越权（跳转执行管理员特有的操作），其根本原因在于应用程序过于信任客户端。
### 如何修复此漏洞

* **实施完善的身份验证与授权检查**：确保系统不仅知道用户是谁，还明确其拥有的权限。
* **验证权限**：在处理每一个请求时，必须实时校验当前已登录的用户是否有权访问其请求的具体资源。
* **使用间接引用或加密令牌**：采用不可预测的随机字符串（如 UUID）或加密 Token，替代易被猜测的自增数字 ID（如 `?id=1`）。
* **日志记录与监控**：对敏感资源的访问尝试进行详细记录并实施监控，以便及时发现异常的越权行为。
## A07：身份验证失败
**身份验证失败（Authentication Failures）** 发生在应用程序无法可靠地验证或绑定用户身份时。常见问题包括：

* **用户名枚举**：可以通过响应差异确认某个用户名是否存在。
* **弱密码/易猜测密码**：且缺乏登录锁定或速率限制机制。
* **登录或注册流程中的逻辑漏洞**：程序处理逻辑不严谨。
* **不安全的会话或 Cookie 处理**：导致会话被劫持或伪造。

如果存在上述任何问题，攻击者通常可以冒充他人登录，或将特定会话绑定到错误的账户上。

## A09：记录&警报失败
当应用程序未能对与安全相关的事件进行记录或警报时，防御者就无法检测或调查攻击行为。良好的日志记录是问责（Accountability）的基础——即能够证明“谁在何时、何地做了什么”。

在实际操作中，此类失效通常表现为：缺失身份验证事件、错误日志含糊不清、对暴力破解或权限变更缺乏警报机制、日志保存时间过短，或者日志存储位置不当导致攻击者可以轻易篡改。

# 应用程序设计缺陷：A02、A03、A04、A06
## A02：安全配置错误

### 1. 定义
**安全配置错误**发生在系统、服务器或应用程序部署时使用了不安全的默认设置、不完整的配置或暴露了不必要的服务。这通常**不是代码漏洞**，而是环境、软件或网络**搭建过程中的失误**，为攻击者提供了便捷的切入点。

### 2. 重要性
即使是微小的配置疏忽也可能导致敏感数据泄露、权限提升，或让攻击者在系统中获得立足点。现代应用依赖复杂的架构、云服务和第三方 API，一个暴露的管理员面板、一个开放的存储桶或配置错误的权限，都可能导致整个系统沦陷。


### 3. 典型案例
2017 年，**Uber** 的一个包含敏感用户数据（包括司机和乘客信息）的 **AWS S3 备份存储桶** 遭到泄露，原因就是该存储桶被设置为公开访问。攻击者无需任何凭据即可直接下载数据。

### 4. 常见表现形式

* **默认凭据**：未修改默认用户名/密码或使用弱密码。
* **不必要的暴露**：将无关紧要的服务或端口直接暴露在互联网上。
* **云存储配置错误**：S3、Azure Blob 或 GCP 存储桶的访问权限设置不当。
* **API 安全缺失**：API 访问不受限，或缺乏必要的身份验证与授权。
* **过度详细的错误信息**：报错页面显示了堆栈轨迹（Stack Traces）或系统底层细节。
* **软件陈旧**：运行着含有已知漏洞的过时软件、框架或容器。
* **AI/ML 端点暴露**：人工智能或机器学习接口缺乏适当的访问控制。

### 5. 防范措施

| 核心策略 | 具体行动 |
| :--- | :--- |
| **系统加固** | 强化默认配置，移除或禁用未使用的功能、模块及服务。 |
| **权限管理** | 在所有系统中强制执行强身份验证和**最小权限原则**。 |
| **网络隔离** | 限制网络暴露面，对敏感资源进行**网络分段**。 |
| **补丁管理** | 确保软件、框架和容器及时更新并打好安全补丁。 |
| **信息隐藏** | 在生产环境的错误消息中隐藏堆栈轨迹和系统详细信息。 |
| **定期审计** | 定期审查云配置和权限设置，确保无越权暴露。 |
| **AI 接口安全** | 使用适当的访问控制和监控手段保护 AI 端点及自动化服务。 |
| **流程自动化** | 将配置审查和自动化安全检查集成到 **CI/CD 部署流水线**中。 |

## A03：软件供应链错误
### 1. 定义
**软件供应链失效**发生在应用程序所依赖的组件、库、服务或模型遭到破坏、过时或未经过妥善验证时。这些弱点并非源于你编写的代码本身，而是存在于你所依赖的软件和工具中。攻击者利用这些薄弱环节植入恶意代码、绕过安全机制或窃取敏感数据。

### 2. 重要性
现代应用程序是由大量的第三方软件包、API 和 **AI 模型**构建而成的。只要有一个依赖项被攻破，就可能导致你的整个系统沦陷，使攻击者在完全不接触你自身代码的情况下获得访问权限。供应链攻击可以自动化且大规模地传播，这使得它们难以检测且极具破坏性。

---

### 3. 典型案例
2021 年的 **SolarWinds Orion** 遭入侵事件展示了供应链攻击的危险性。攻击者将恶意代码植入到一个受信任的更新补丁中，导致成千上万个自动安装该更新的机构受到影响。这并非 SolarWinds 核心逻辑中的漏洞，而是**软件更新构建、验证及分发过程**中的缺陷。

在 **AI 领域**，如果我们使用未经核实的第三方模型或微调数据集，也可能观察到此类问题——这些模型或数据可能嵌入了隐藏行为、后门或偏见输出，从而导致系统受损或数据泄露。

---

### 4. 常见表现形式 
* **使用未经验证或维护的库**：直接使用来源不明或已停止维护的依赖项。
* **自动更新风险**：在没有进行安全验证的情况下，自动安装第三方更新。
* **过度依赖 AI 模型**：在缺乏监控或审计的情况下，过度信赖第三方 AI 模型。
* **构建流水线不安全**：CI/CD 流程存在缺陷，允许攻击者篡改构建过程。
* **溯源追踪缺失**：对组件的许可证或来源追踪不力。
* **部署后缺乏监控**：在应用上线后，未能持续监控依赖项中新出现的漏洞。

---

### 5. 如何保护供应链 
| 核心策略 | 具体行动 |
| :--- | :--- |
| **准入验证** | 在使用所有第三方组件、库和 AI 模型之前进行严格审核。 |
| **持续修补** | 定期监控依赖项的状态并及时修补已知漏洞。 |
| **数字签名** | 对软件更新和软件包进行签名、验证及审计。 |
| **锁定流水线** | 严密保护 CI/CD 流水线和构建流程，防止未经授权的篡改。 |
| **溯源管理** | 跟踪所有依赖项的来源（Provenance）和许可协议。 |
| **运行监控** | 实施运行时监控，以发现依赖项或 AI 组件产生的异常行为。 |
| **集成安全开发** | 将供应链威胁建模整合到**软件开发生命周期 (SDLC)** 中，涵盖测试、部署及更新工作流。 |

## A04：加密失败
### 1. 定义 
**加密失败**发生在加密技术被错误使用或完全未被使用的情况下。这包括使用弱加密算法、硬编码密钥、密钥处理不当或对敏感数据不加加密。这些缺陷使得攻击者能够访问本应处于保护状态的私密信息。

### 2. 重要性 
Web 应用程序处处依赖加密技术：保护网络流量、确保存储数据安全、验证身份以及守护机密信息。当这些控制措施失效时，密码、令牌或个人信息等敏感数据就会暴露，导致账户被接管或发生大规模的数据泄露。

攻击者可以通过**中间人攻击 (MITM)**、对弱密钥进行**暴力破解**，或者简单地发现从未得到妥善保护的密钥来利用这些缺陷。

---

### 3. 常见表现形式 

* **使用过时或过弱的算法**：例如 MD5、SHA-1 或 DES/ECB 模式。
* **硬编码机密信息**：将密钥、密码直接写在代码或配置文件中。
* **密钥管理不善**：缺乏密钥轮换机制或管理流程混乱。
* **缺乏加密保护**：对传输中（Transit）或存储中（Rest）的敏感数据未进行加密。
* **证书问题**：使用自签名或无效的 TLS 证书。
* **AI/ML 系统安全风险**：在处理模型参数或敏感输入时，缺乏妥善的机密信息处理机制。

---

### 4. 防范措施

| 核心策略 | 具体行动 |
| :--- | :--- |
| **升级算法** | 使用强效的现代算法（如 **AES-GCM**、**ChaCha20-Poly1305**），并强制执行带有效证书的 **TLS 1.3**。 |
| **专业管理** | 使用专业的密钥管理服务，如 Azure Key Vault、AWS KMS 或 HashiCorp Vault。 |
| **定期轮换** | 遵循定义的加密周期，定期轮换机密信息和密钥。 |
| **制度化管理** | 制定并强制执行密钥生命周期管理的标准操作程序 (SOP) 和政策。 |
| **资产清单** | 维护证书、密钥及其所有者的完整清单。 |
| **AI 安全保障** | 确保 AI 模型和自动化代理永远不会暴露未经加密的机密或敏感数据。 |

## A06：不安全的设计

### 1. 定义 
**不安全的设计**是指在系统构建之初，由于逻辑缺陷或架构错误而产生的问题。这些缺陷通常源于跳过了威胁建模、缺乏设计需求或评审，或者是由于偶然的失误。

此外，随着 **AI 助手**的引入，AI 系统加剧了不安全设计的风险。开发者往往假设模型是安全、准确且可预测的，或者认为 AI 生成的代码是无瑕疵的。当 AI 系统能够无限制地生成查询、编写代码或对用户分类时，风险就深深植入了设计之中，导致架构模式的脆弱。

### 2. 案例
**Clubhouse** 是一个很好的例子。其早期设计假设用户只会通过移动应用进行交互，但后端 **API 却缺乏适当的身份验证**。任何人都可以直接查询用户数据、房间信息甚至是私人对话。当研究人员对其进行测试时，整个“私人对话”的前提瞬间崩塌。

### 3. 重要性
**不安全的设计是无法通过“打补丁”来修复的。** 它已经根植于工作流、逻辑和信任边界之中。修复它意味着需要重新构思系统（以及现在的 AI）做出决策的方式。

---

### 4. 2025 年常见的不安全设计
* **业务逻辑控制薄弱**：例如找回密码或审批流程存在逻辑漏洞。
* **错误的假设**：对用户行为或 AI 模型行为抱有不切实际的完美假设。
* **过度授权的 AI 组件**：AI 拥有未受限制的权限或访问能力。
* **缺乏护栏**：大语言模型（LLM）和自动化代理缺乏安全防御措施。
* **遗留调试接口**：在生产环境中保留了测试或调试用的绕过机制。
* **忽视滥用案例分析**：未进行 AI 威胁建模或一致的滥用场景评审。

---

### 5. AI 时代的挑战
* **提示词注入 (Prompt Injection)**：用户输入与系统提示词混淆，导致攻击者劫持上下文或窃取隐藏数据。
* **盲目信任模型输出**：系统在没有人工验证的情况下直接根据 AI 决策执行操作，极其脆弱。
* **模型投毒 (Model Poisoning)**：从不可信来源获取或使用不安全数据微调的模型，可能埋下隐藏的后门。

---

### 6. 如何实现安全设计
* **零信任 AI**：除非证明其安全，否则视所有模型为不可信。
* **双向校验**：对模型的所有输入和输出进行验证和过滤。
* **逻辑隔离**：将系统提示词与用户内容完全分离。
* **权限最小化**：对用户、API 和服务应用最小权限原则。
* **人工介入**：高风险的 AI 操作必须经过人工复核。
* **全程建模**：将威胁建模贯穿开发的每个阶段，而不仅仅是开始阶段。

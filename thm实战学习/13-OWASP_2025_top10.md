**OWASP Top 10 2025** 是由开放式 Web 应用程序安全项目（OWASP 基金会）发布的最新版**全球十大 Web 应用安全风险报告**。

OWASP 每几年会根据行业调查和漏洞数据更新一次这份名单。2025 版反映了现代开发环境（如云原生、AI 驱动和复杂的供应链）中安全威胁的变化。

# 2025 年十大安全风险列表

| 排名 | 风险类别 | 核心关注点 |
| :--- | :--- | :--- |
| **A01** | **失效的访问控制 (Broken Access Control)** | 依然稳居榜首。指用户越权访问了不该看的数据或功能（如普通用户查看管理员后台）。 |
| **A02** | **安全配置错误 (Security Misconfiguration)** | 排名大幅上升。涵盖了不安全的默认设置、云存储权限泄露、多余的 HTTP 标头等。 |
| **A03** | **软件供应链失败 (Software Supply Chain Failures)** | **新增/重大更新**。关注第三方库、CI/CD 流水线和包管理器的安全（如恶意组件注入）。 |
| **A04** | **加密失败 (Cryptographic Failures)** | 保护敏感数据的加密手段不足、使用过时的加密算法或密钥泄露。 |
| **A05** | **注入 (Injection)** | 排名有所下滑。包括 SQL 注入、跨站脚本 (XSS)、OS 命令注入等，主要源于对输入过滤不严。 |
| **A06** | **不安全的设计 (Insecure Design)** | 强调“安全应始于设计阶段”。如果架构本身有问题，后续写再好的代码也补不回来。 |
| **A07** | **身份验证失败 (Authentication Failures)** | 登录流程中的缺陷，例如允许弱密码、凭据填充攻击或会话管理不当。 |
| **A08** | **软件或数据完整性失败 (Integrity Failures)** | 应用程序在没有验证签名或完整性的情况下加载了不可信的代码、数据或对象。 |
| **A09** | **安全日志和记录失败 (Logging & Alerting Failures)** | 攻击发生时没有记录日志，或者报警太慢，导致黑客在系统中潜伏数月无人知晓。 |
| **A10** | **异常情况处理不当 (Mishandling of Exceptional Conditions)** | **2025版新入选**。指系统在报错时泄露敏感信息，或因未捕获异常导致拒绝服务或逻辑绕过。 |

# IAAA失败：A07、A07、A09
## IAAA
**IAAA** 是思考用户及其操作在应用程序中如何被验证的一种简单方式。其中的每一个环节都至关重要，且无法跳过任何一个层级。这意味着，如果前一个环节没有执行，你就无法进行后续的操作。这四个项分别是：

* **标识 (Identity)** —— 代表个人或服务的唯一账号（例如：用户 ID 或电子邮件）。
* **认证 (Authentication)** —— 证明该身份的真实性（例如：密码、一次性密码 OTP、通行密钥 Passkeys）。
* **授权 (Authorization)** —— 确定该身份被允许执行的操作。
* **审计/问责 (Accountability)** —— 记录并警报谁在何时、何地做了什么。

本节讨论的 **OWASP Top 10:2025** 中的三个类别，都与 **IAAA** 实施过程中的失败有关。这些环节的弱点可能具有极大的破坏性，因为它们可能允许威胁参与者访问其他用户的数据，或获得超出其应有范围的更高权限。

## A01：失效的访问控制
 失效的访问控制（Broken Access Control）发生在服务器未能针对每一个请求正确执行“谁能访问什么”的限制时。这种情况最常见的案例是 IDOR（不安全的直接对象引用）：如果你通过修改 ID（例如将 ?id=7 改为 ?id=6）就能查看或编辑他人的数据，那么访问控制就失效了。

在实际操作中，这通常表现为水平越权（相同角色，访问其他用户的数据）或垂直越权（跳转执行管理员特有的操作），其根本原因在于应用程序过于信任客户端。
### 如何修复此漏洞

* **实施完善的身份验证与授权检查**：确保系统不仅知道用户是谁，还明确其拥有的权限。
* **验证权限**：在处理每一个请求时，必须实时校验当前已登录的用户是否有权访问其请求的具体资源。
* **使用间接引用或加密令牌**：采用不可预测的随机字符串（如 UUID）或加密 Token，替代易被猜测的自增数字 ID（如 `?id=1`）。
* **日志记录与监控**：对敏感资源的访问尝试进行详细记录并实施监控，以便及时发现异常的越权行为。
## A07：身份验证失败
**身份验证失败（Authentication Failures）** 发生在应用程序无法可靠地验证或绑定用户身份时。常见问题包括：

* **用户名枚举**：可以通过响应差异确认某个用户名是否存在。
* **弱密码/易猜测密码**：且缺乏登录锁定或速率限制机制。
* **登录或注册流程中的逻辑漏洞**：程序处理逻辑不严谨。
* **不安全的会话或 Cookie 处理**：导致会话被劫持或伪造。

如果存在上述任何问题，攻击者通常可以冒充他人登录，或将特定会话绑定到错误的账户上。

## A09：记录&警报失败
当应用程序未能对与安全相关的事件进行记录或警报时，防御者就无法检测或调查攻击行为。良好的日志记录是问责（Accountability）的基础——即能够证明“谁在何时、何地做了什么”。

在实际操作中，此类失效通常表现为：缺失身份验证事件、错误日志含糊不清、对暴力破解或权限变更缺乏警报机制、日志保存时间过短，或者日志存储位置不当导致攻击者可以轻易篡改。

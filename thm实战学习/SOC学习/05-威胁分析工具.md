# 网络威胁情报（CTI）简介

## 认识CTI
在 SOC 环境下，CTI 是将“报警噪音”转化为“决策依据”的上下文。它旨在回答三个核心问题：
1. **谁（或什么）**在报警指标的另一端？
2. 他们过去的**行为特征**是什么？
3. 我的组织该**如何响应**？（现在该做什么？）

---

### 数据进阶之路：从原始数据到情报
安全分析师的任务就是通过**富化（Enrichment）**，将零散的数据提升为可用的情报。

| 层级 | 定义 | 例子（以 IP 为例） | SOC L1 动作 |
| :--- | :--- | :--- | :--- |
| **数据 (Data)** | 未处理的观察值 | `45.155.205.3:443` | 捕获原始证据 |
| **信息 (Information)** | 数据 + 事实标注 | 该 IP 注册于 Hetzner，首次发现于 2023 年 | 记录属性 |
| **情报 (Intelligence)** | 经过分析，回答“那又怎样” | 该 IP 属于 BumbleBee 僵尸网络的 C2；**立即封禁** | 上报或处置 |

---

### 三大关键指标类型
* **失陷指标 (IOC)：** 已被攻破的证据（如日志中的恶意 C2 地址）。
* **攻击指标 (IOA)：** 正在发生的恶意动作（如 PowerShell 启动了一个未知服务）。
* **战术、技术与过程 (TTP)：** 攻击者的详细方法论（对应 MITRE ATT&CK ID）。

---

### 常见指标富化指南 (实用工具清单)

| 指标类型 | 示例 | 推荐查询资源 |
| :--- | :--- | :--- |
| **IP 地址** | `45.155.205.3` | WHOIS (ASN 信息)、VirusTotal、Shodan (端口扫描) |
| **域名** | `malicious-updates.net` | WHOIS 注册日期、RiskIQ (被动 DNS)、urlscan.io |
| **URL** | `hxxp://.../login` | URLhaus (信誉库)、Any.Run (沙箱动态运行) |
| **文件哈希** | `e99a1...` | VirusTotal、Hybrid-Analysis |
| **本地注册表** | `HKCU\Software\Run\...` | Sigma 规则、EDR 普查、厂商知识库 |

---

### 馈送 (Feed) vs 平台 (Platform)
* **馈送 (Feed)：** 指标的定期流（CSV/JSON 格式）。注意：盲目引入 Feed 会导致误报淹没分析师。
* **平台 (Platform)：** 存储、富化和追踪指标的仓库。代表工具：**MISP** (开源) 和 **OpenCTI**。

---

### 威胁情报的四个来源
1.  **内部遥测 (Internal)：** 组织自己的 SIEM/EDR 日志。**关联度最高。**
2.  **商业服务 (Commercial)：** 付费的情报 Feed 或沙箱。**准确度高，但有授权限制。**
3.  **开源情报 (OSINT)：** 公开博客、AbuseIPDB。**需交叉验证真伪。**
4.  **行业社区 (ISACs)：** 特定行业（如航空、金融）共享的敏感情报。

---

### 威胁情报的分类 (Classifications)

* **战略情报 (Strategic)：** 面向高层。关注长期趋势、行业风险（如：医疗行业勒索软件趋势报告）。
* **战术情报 (Tactical)：** 面向技术人员。分析攻击者的 **TTP**（如：详细说明某种利用 Visual Basic 传播木马的方法）。
* **运营情报 (Operational)：** 针对特定攻击活动。关注攻击者的动机、意图以及组织内哪些资产可能被盯上。
* **技术情报 (Technical)：** 最基础的原子级指标，如 IP、哈希值等。**这是 L1 分析师处理最多的部分。**
## 网络威胁情报（CTI）生命周期


### 一、 核心概念：分享协议与数据格式

在进入生命周期之前，必须掌握两个基础工具：

#### 1. TLP 流量灯协议 (Traffic Light Protocol)
由 FIRST.org 定义，规定了情报的**共享边界**。
* **TLP: CLEAR (白色/透明)**：无限制，可发至内网维基。
* **TLP: GREEN (绿色)**：仅限同行社区（如合作伙伴 SOC），不可公开。
* **TLP: AMBER (琥珀色)**：仅限组织内部及“有必要知晓”的客户。
* **TLP: RED (红色)**：仅限特定人员，必须加密存储，严禁进入常规工单系统。
    > **原则**：如果同一指标有多个标签，遵循**最严格**的一个。

#### 2. 情报格式：STIX
* **定义**：结构化威胁信息表达（Structured Threat Information Expression）。
* **本质**：一种 **JSON 模式**，旨在以机器可读的方式描述威胁指标、关系及上下文。

---

### CTI 生命周期：实战演练

**场景**：Alex 需要保护公司核心的 PostgreSQL 生产数据库。

### 第一步：方向 (Direction) —— 明确使命
* **核心任务**：将高层的“引入情报”指令转化为具体的、可衡量的需求。
* **关键资产**：PostgreSQL 数据库。
* **产出问题**：
    1.  哪些外部 IP/域名正在利用 PostgreSQL 漏洞或尝试拖库？
    2.  本周活跃的针对数据库凭据的木马家族及其文件哈希是什么？

### 第二步：收集 (Collection) —— 汇聚素材
Alex 从四个渠道获取原始数据：
* **商业馈送**：防火墙厂商提供的最新 37 个 C2 地址。
* **开源情报 (OSINT)**：从 AbuseIPDB 提取的暴力破解 IP。
* **内部平台 (MISP)**：历史事件中的凭据窃取木马哈希。
* **厂商报告**：本周发布的战略洞察及关联的 IOC。

### 第三步：处理 (Processing) —— 规格化与关联
原始数据格式杂乱，需要通过 Python 脚本进行“清洗”：
* **规格化**：统一 IPv6 压缩格式、域名小写化、去除子网掩码。
* **关联与去重**：对比现有数据库，剔除重复项。
* **转换格式**：生成防火墙直接可用的 `blocklist.csv` 和 EDR 可用的 `YARA` 规则。

### 第四步：分析 (Analysis) —— 转化为判断
**拒绝盲目封禁**。Alex 对指标进行分级：
* **高置信度**：在 ≥2 个来源中出现，且在本地日志（Splunk）中确实发现过其探测记录。**（处理建议：立即封禁）**
* **中置信度**：单一可信来源，本地无记录。**（处理建议：仅告警）**
* **低置信度**：仅限 OSINT 来源，无上下文。**（处理建议：监控 14 天）**

### 第五步：分发 (Dissemination) —— 送达消费者
情报必须适配受众的口味：
* **防火墙团队**：接收 CSV 封禁列表。
* **终端团队**：接收 YARA 规则集并加载至 EDR 策略。
* **管理层**：接收 200 字的简报，说明流程的**投资回报率 (ROI)**。

### 第六步：反馈 (Feedback) —— 评估与优化
两周后评估关键绩效指标 (KPI)：
* **停留时间 (Dwell Time)**：由于开启了预先封禁，暴力破解 IP 的停留时间从 48 小时降至 **0 小时**。
* **误报率**：保持在 **0%**。
* **结果**：管理层批准扩大项目，下一阶段将加入 DNS 隧道检测指标。

## CTI标准与框架

### 1. MITRE ATT&CK：攻击者的“罗塞塔石碑”
ATT&CK 将零散的攻击行为标准化。
* **作用**：为技术打上通用标签（如 `T1059.001` 代表 PowerShell），确保不同厂商、团队和审计员之间沟通无障碍。
* **L1 分析师用法**：
    1.  将警报行为匹配到具体的战术/技术 ID。
    2.  在工单中记录 ID（例如：“针对财务服务器发现 `T1071.001` Web 远程控制行为”）。
    3.  移交 L2/IR 团队，他们能立即根据 ID 匹配防御方案。

### 2. MITRE D3FEND：防御者的“对策百科”
如果说 ATT&CK 是“矛”，D3FEND 就是“盾”。
* **作用**：为每项攻击技术匹配具体的防御措施（如凭据加固、数据混淆）。
* **实战举例**：
    1.  发现 DNS 隧道攻击 (`T1048.003`)。
    2.  查询 D3FEND 匹配项：`D3-NTDN`（DNS 请求分析）。
    3.  **动作**：建议阻断超长 TXT 记录或针对异常查询熵值报警。

### 3. 网络杀伤链 (Cyber Kill Chain)：攻击的七个阶段
由洛克希德·马丁公司开发，将攻击分解为线性步骤，帮助分析师判断攻击所处的阶段。

| 阶段 | 目的 | 例子 |
| :--- | :--- | :--- |
| **侦察 (Recon)** | 获取受害者信息 | 收集邮箱、OSINT、网络扫描 |
| **武器化 (Weaponisation)** | 制作恶意软件 | 将后门打包进 Office 文档 |
| **投递 (Delivery)** | 传输恶意程序 | 邮件附件、恶意链接、USB |
| **漏洞利用 (Exploitation)** | 执行代码 | 利用 EternalBlue 漏洞进入系统 |
| **安装 (Installation)** | 建立持久化 | 安装后门、密码窃取工具、RAT |
| **命令与控制 (C2)** | 远程控制 | 使用 Cobalt Strike 等控制被指控系统 |
| **行动目标 (Actions)** | 达成最终目的 | 加密数据（勒索）、窃取商业机密 |

### 4. 漏洞识别：CVE, CVSS 与 NVD
SOC 队列中充斥着漏洞通知，分析师需要通过以下标准识别优先级：
* **CVE** (通用漏洞披露)：漏洞的“身份证号”，如 `CVE-2023-4863`。
* **CVSS** (通用漏洞评分系统)：0-10 分的严重性等级，分值越高越危险。
* **NVD** (国家漏洞数据库)：官方仓库，将 CVE、评分、漏洞利用代码和受影响产品关联在一起。

### 5. 情报的分享与传输：STIX 与 TAXII
这是情报自动化的两大支柱：
* **STIX (结构化威胁信息表达)**：一种 **JSON 语言格式**，描述“威胁长什么样”。
* **TAXII (指标信息可信自动化交换)**：一套 **API 传输协议**，解决“情报怎么传”。
    * **Collection (集合模式)**：生产者托管，消费者按需拉取。
    * **Channel (频道模式)**：中心服务器主动推送给订阅用户。

# 文件和哈希威胁情报
## 文件名和路径

### 1. 文件路径分析：攻击者的“藏身之处”
文件路径是分析师最先接触到的信息。攻击者会选择特定的路径来避开监控或获取权限。

#### 核心路径及其风险：
* **`C:\` (系统根目录)**：常被用于设置**持久化**机制。
* **`C:\Users\Public` (公共用户配置文件)**：这是一个“全家桶”目录。攻击者在这里投放工具，可以让这台电脑上的**所有用户**都能访问和运行，方便横向移动。
* **`C:\Users\Public\Public Downloads`**：由于该目录平时文件流动频繁，安全审计往往不够严格，容易躲过监测。
* **`C:\Windows\Temp\` (临时目录)**：用于存放**临时载荷（Ephemeral Payloads）**。攻击者完成任务后会在这里清理痕迹，因为系统清理临时文件是常态。
* **`C:\ProgramData\`**：这是一个隐藏的、对许多程序具有写权限的系统路径。攻击者在这里放置载荷，可以实现**隐蔽的持久化**。

---

### 2. 文件名启发式指标：攻击者的“变装术”
攻击者通过修改文件名来降低用户的警惕性，或绕过简单的关键词检测。

| 技巧类型 | 示例 | 原理解析 |
| :--- | :--- | :--- |
| **双重扩展名** | `invoice.pdf.exe` | 利用 Windows 默认“隐藏已知文件扩展名”的设置。用户在屏幕上只看到 `invoice.pdf`，点开却是恶意程序。 |
| **系统程序伪装** | `scvhost.exe` | 模仿核心系统进程（如 `svchost.exe`）。利用分析师对系统文件名的熟悉感进行欺骗。**（注意：必须结合路径判断，不在 `System32` 里的 `svchost` 必有问题）** |
| **高熵字符串** | `jh8F21.exe` | 文件名是一串随机乱码。这通常意味着该程序是自动生成的、加壳的或者是钓鱼邮件批量产生的，不符合人类命名习惯。 |
| **伪装混淆** | `backup-2300.exe` | 起一个看起来人畜无害的名字，混入日常备份文件中。或者是使用单字符替换（如字母 `o` 换成数字 `0`）来误导肉眼。 |

## 文件哈希值

### 1. 文件哈希：数字指纹
文件名和路径可以随时更改，但**哈希值（如 SHA256, MD5）**是二进制文件的唯一指纹。只要文件内容改变一个字节，哈希值就会完全不同。

**生成命令：**
* **Windows (CMD):** `certutil -hashfile 文件名 SHA256`
* **Windows (PowerShell):** `Get-Filehash -Algorithm SHA256 文件名`
* **Linux:** `sha256sum 文件名`

**分析师准则：**
* 统一使用**小写**字母存储哈希，避免对比差异。
* **全链条哈希**：如果恶意软件在 ZIP 里，既要对压缩包哈希，也要对解压后的文件哈希。
* **上下文记录**：记录发现该哈希的时间、地点，不要只留一串字符。

---

### 2. 使用 VirusTotal (VT) 进行分析
VirusTotal 聚合了数十家厂商的扫描结果。在分析时，应重点关注以下维度：

| 关注维度 | 关键问题 & 红旗信号 (Red Flags) | 分析师思考 |
| :--- | :--- | :--- |
| **检出率与标签** | **5 个以上**知名厂商标记为恶意。 | 关注家族名称（如 Trojan, Ransomware）。新病毒初期检出率可能很低。 |
| **上传时间** | 首次上传时间及检出数增长趋势。 | 厂商通常需要 48-72 小时完成深度分析。 |
| **数字签名** | 签名无效、缺失或签给无关实体。 | 合法签名也可能被窃取或滥用，检查证书链有效期。 |
| **文件属性** | 编译时间异常（如凌晨 3 点）；**高熵值 (>7.5)**。 | 高熵通常意味着文件被加壳或加密。 |
| **关联信息 (Relations)** | 连接到已知恶意 IP 或 DGA 域名。 | 检查 IP 在 Shodan 上的端口，警惕类似随机生成的域名。 |
| **行为特征 (Behavioral)** | 修改关键注册表、进程注入。 | 需结合本地端点日志（EDR）进行关联分析。 |

---

### 3. 使用 MalwareBazaar 进行深度狩猎
相比 VirusTotal，**MalwareBazaar** 更侧重于恶意软件的收集与样本共享。

* **功能特点：**
    * **恶意软件家族标签：** 即使 VT 检出率只有 5/70，如果被标记了 `#IcedID`，也应视为恶意。
    * **YARA 规则集成：** 提供的规则可直接导入 EDR/SIEM 进行狩猎。
    * **攻击者归属 (Campaign)：** 标签如 `#TA551` 能将孤立事件关联到已知的黑客组织。
    * **样本获取：** 提供恶意软件样本下载，供分析师在沙箱中重放和分析。

## 沙箱分析

### 什么是沙箱分析？
静态属性（如哈希、字符串）只能告诉你文件的“身份”，而**动态分析**通过在受控的隔离环境（虚拟机）中实际运行恶意软件，来观察它的“行为”。

#### 沙箱的三大核心作用：
1.  **确认执行**：验证文件运行后是否真的有动作。如果没反应，告警可能是虚警。
2.  **提取运行时 IOC**：抓取它连接的域名、创建的互斥量（Mutex）、释放的后续载荷（Payload）。
3.  **映射 ATT&CK**：大多数沙箱会自动将行为贴上 MITRE ATT&CK 技术 ID 标签，加速报告编写。

---

## 2. 主流沙箱工具对比

| 工具名称 | 侧重点 | 适用人群/场景 |
| :--- | :--- | :--- |
| **Hybrid Analysis (HA)** | 行为树、清晰的 ATT&CK 热力图 | 快速生成执行摘要、初级分析师 |
| **Joe Sandbox (JS)** | 深度分析：系统调用、内存转储、字符串提取 | 逆向工程师、检测工程师 |

---

## 3. 沙箱报告的关键内容
当你拿到一份 100/100 威胁分数的报告时，应关注：
* **风险行为**：例如修改敏感注册表、自我删除、注入进程。
* **文件构成**：导入导出函数、文件大小、编译时间。
* **网络连接**：它尝试联络的 IP 地址和域名。
* **进程链**：父进程启动了哪些子进程（如 `cmd.exe` 启动了 `powershell.exe`）。

---

## 4. 沙箱的局限性（重点：为什么不能迷信沙箱）
攻击者会专门设计手段来欺骗沙箱，导致出现 **False Negatives（漏报/假阴性）**。

### A. 沙箱逃逸技术 (Evasion)
* **环境察觉**：恶意软件会检查自己是否在虚拟机里（如检查显卡驱动、硬盘序列号）。
* **反调试/反沙箱**：检测是否存在调试器。

### B. 执行时间限制
* 大多数沙箱只运行 2-5 分钟。
* **对抗手段**：攻击者会设置**延迟执行**（比如运行后等 10 分钟再发作），或者需要多步交互才能触发。

### C. 加密与混淆流量
* 沙箱往往无法解密 SSL/TLS 流量，导致无法看到 HTTPS 通信的具体内容。
* 利用 **DNS 隧道 (DNS Tunnelling)** 将数据藏在 DNS 查询中。

### D. 无文件攻击与 LotL (白加黑)
* 有些攻击只在内存中运行，不产生磁盘文件（Fileless）。
* 利用系统自带工具（如 PowerShell、WMI）执行任务，沙箱可能视其为合法行为。

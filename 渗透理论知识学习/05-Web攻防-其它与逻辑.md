# XXE漏洞
全称为XML Enternal Entity Injection，中文名称为：XML外部实体注入。

- 漏洞成因：解析时未对XML外部实体加以限制，导致攻击者将恶意代码注入到XML中，导致服务器加载恶意的外部实体引发文件读取，SSRF，命令执行等危害操作。

- 特征：在HTTP的Request报文出现以下请求报文，即表明此时是采用XML进行数据传输，就可以测试是否存在XML漏洞。

    `Content-type:text/xml application/xml`
- 简单了解XML：

    1. XML 指可扩展标记语言（EXtensible Markup Language）
    2. XML 是一种标记语言，很类似 HTML
    3. XML 被设计为传输和存储数据，其焦点是数据的内容 XML 被设计用来结构化、存储以及传输信息
    4. XML 允许创作者定义自己的标签和自己的文档结构

漏洞利用方式和SQL注入很像，也分有无回显，有回显的情况可以直接在页面中看到payload的执行结果或现象，无回显的情况又称为 blind xxe（类似于布尔盲注、时间盲注），可以使用外带数据(OOB)通道提取数据。（遇到的时候再具体学）。
# SSRF服务器端请求伪造
server-side request forgery，是一种由攻击者构造形成，由服务端发起请求的一个安全漏洞。一般情况下，SSRF的攻击目标是从外网无法访问的内部系统。

- 成因：服务器端提供了从其他服务器应用获取数据的功能且没有对目标地址做过滤和限制。

## 漏洞利用
1. 获得出口IP

    比如，可以获得百度翻译的出口IP。百度翻译的功能：它允许你输入一个 URL，然后它的服务器会爬取那个页面的内容并翻译给你看。可以让百度翻译翻译我们服务器上的test.txt，然后查看服务器的日志，就可以获得其出口IP。
2. 获取本地文件信息：可以直接配合FILE协议进行本地信息的获取。
3. 获取内网Web资产信息：如果通过某种手段知道了公司内网里有哪些秘密网站的地址（或者 IP），就可以利用 SSRF 漏洞，去读取这些秘密网站的内容。因为请求是从公司内部发出的，内网的防火墙通常不会拦截。
4. 探测内网端口信息：这里的端口必须带banner回显才可以探测出来。

## SSRF中URL的伪协议
当我们发现SSRF漏洞后，首先要做的事情就是测试所有可用的URL伪协议
```
file:// 从本地文件系统中获取文件内容，如，file://etc/passwd
dict:// 字典服务器协议，访问字典资源，如，dict://ip:6739/info
sftp:// SSH文件传输协议或安全文件传输协议
ldap:// 轻量级目录访问协议
tftp:// 简单文件传输协议
gopher:// 分布式文档传递服务，目前已经慢慢淡出历史，可使用gopherus生成payload
http:// 超文本传输协议，基于TCP/IP，利用如:http://serverip:port/
```
# 业务逻辑漏洞
1. 验证码安全
    
    作用：防止攻击者进行暴力破解，防止机器人进行自动注册、登录、灌水等。

    但是，验证码也有很大的安全风险。
    1. 验证码过于简单：容易被验证码识别引擎或者接口识别出来，依旧可以用机器进行批量操作。
    2. 数字暴力破解：如果验证码是4位的，可以尝试暴力破解。Burpsuite爆破0000-9999开启多线程爆破只需要5分钟左右。
    3. 验证码不失效：验证码被错误设定为根据用户是否发起新的验证码请求来决定是否失效，而不是使用1次后就失效。可以使用burpsuite抓包然后不放，用SQLMap进行注入测试。
    4. 验证码直接返回：抓包可以看到验证码在回显的HTTP相应包中显示。验证码在客户端中直接返回，说明是把网站的验证功能放在了客户端来进行校验，过分信任客户端，从而产生漏洞。
    5. 验证码首次绕过：若网站设置了第一次登录无需验证码，第一次登录失败时才需要验证码，则可以通过尝试来判断是什么让服务端认为是“第一次登录”（比如session信息），进而模拟第一次登录的数据包，来绕过验证码。

2. 支付漏洞
    1. 金额数据直接修改：网站在支付时直接使用前端传过来的金额，并且没有对金额进行验证，导致金额数据篡改的产生。正常操作应该是在后端计算订单金额。
    2. 商品数量修改：与上面差不多，前端限制数量不能为0，但是后端没有做校验，买一个20块的和-19个1块的就能用1块买到20块的产品。可以绕过上面的后端计算订单金额。
    3. 运费/优惠券金额修改：跟上面一样，如果上面两个都做了校验，可以来尝试一下这个漏洞是否存在。

3. 账户越权

    Web应用系统的权限校验存在问题，攻击者就可以通过这些问题来访问未经授权的功能或数据。
    1. 未授权：开发者未考虑到用户是否经过登录或者认证的情况下直接返回敏感数据。比如`http://x.x.x.x/usrinfo.php?id=1`如果没有做验证，那么所有人都可以访问到，将导致用户信息被遍历输出。
    2. 水平越权：攻击者尝试访问与他拥有相同选前的用户的资源，比如其他用户信息、订单等。比如查询自己订单信息时，发现请求头是`...api?GoodsOrderid=1`，修改最后的参数，发现返回了别人的订单。
    3. 垂直越权：低权限攻击者访问高权限用户的资源。比如若程序对用户的权限控制仅仅是让用户界面不出现相应的菜单及功能模块，那么我们可以尝试修改菜单ID来实现垂直越权。

4. 密码重置

    密码找回功能有漏洞，导致其它普通用户也可以重置任意用户的密码。
    1. 验证码不失效：可以使用burpsuite暴力破解，现在已经很少见了。
    2. 验证码直接返回：本质是在客户端进行验证码比对。流程是：输入手机号后点击获取验证码，验证码在客户端生成，并直接返回HTML文件中或者网页的响应包中，以便接下来进行比对。（有的甚至直接返回了密码相关的值）
    3. 跳过验证步骤：用户可以直接输入最终修改密码的网址，跳转到该页面。（挖掘该漏洞需要用自己的账号走一遍完整流程，记下关键链接或数据包）
    4. 手机号未绑定用户：系统没有将用户名、手机号、验证码三者进行同一的验证。例如：点击找回密码，首先输入其他人的用户名，点击获取验证码，bp抓包把手机号改为我们自己的手机号，然后输入我们获得的验证码，尝试是否能够跳到重置密码的界面。
    5. 修改密码ID替换：修改密码的时候，并没有对原密码进行判断，并且直接根据用户的ID值来修改用户的密码。

        类似SQL语句：`update user set password="1234" where id ='1'`

        这种情况下修改数据包里的ID值，即可修改目标ID锁对应的用户的密码。（这种漏洞也需要尝试才能发现，并且一般同时存在SQl注入漏洞）
